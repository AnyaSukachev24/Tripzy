# Tripzy Evaluation Framework Guide

## Overview

The Tripzy evaluation framework uses an **LLM-as-Judge** approach to assess the quality of travel plans generated by the agent. This guide explains how the evaluation system works, how to use it, and how to add new test cases.

## Why LLM-as-Judge?

Traditional rule-based evaluation is brittle and difficult to maintain for complex, open-ended tasks like travel planning. LLM-as-Judge offers:

- **Flexibility**: Evaluates natural language outputs without rigid rules
- **Comprehensive**: Assesses multiple dimensions (cost realism, preference alignment, conversation quality)
- **Adaptable**: Easy to add new evaluation criteria
- **Explainable**: Provides reasoning for scores

## Evaluation Dimensions

The framework evaluates 7 key dimensions:

### 1. Cost Realism Evaluation

**What it checks:**
- Are accommodation prices realistic for the destination and tier?
- Are activity costs reasonable?
- Are meal costs appropriate?
- Is the total budget realistic for the duration?

**Score:** 0-1, where 1 = perfectly realistic costs

**Use case:** All test cases with budget constraints (most plans)

**Example output:**
```json
{
  "score": 0.85,
  "reasoning": "Accommodation costs are appropriate for mid-range Barcelona hotels. Activity prices are realistic. However, the $150 museum pass seems inflated.",
  "issues": ["Museum pass overpriced"]
}
```

### 2. Preference Alignment Evaluation

**What it checks:**
- Do activities match the trip type (honeymoon, family, adventure, etc.)?
- Is accommodation tier suitable?
- Are expected activities included?
- Is daily pacing appropriate?

**Score:** 0-1, where 1 = perfect alignment with preferences

**Use case:** All test cases with trip type and activity expectations

**Example output:**
```json
{
  "score": 0.9,
  "reasoning": "Excellent honeymoon itinerary with romantic dinners, spa treatments, and private beach time. Luxury accommodation appropriate.",
  "matched_preferences": ["romantic dining", "spa", "private beach"],
  "missed_preferences": ["sunset cruise"]
}
```

### 3. Multi-Turn Conversation Evaluation

**What it checks:**
- Did the agent ask appropriate clarifying questions?
- Did information gathering flow naturally?
- Was context remembered between turns?
- Were critical questions asked?

**Score:** 0-1, where 1 = perfect conversation handling

**Use case:** Test cases with `conversation_turns` field

**Example output:**
```json
{
  "score": 0.75,
  "reasoning": "Agent asked about budget and duration appropriately. However, missed asking about specific interests.",
  "appropriate_questions_asked": ["What's your budget?", "How long can you travel?"],
  "missing_questions": ["What activities interest you?"],
  "conversation_quality": "Good flow but could be more thorough"
}
```

### 4. Edge Case Handling Evaluation

**What it checks:**
- Did the agent identify impossible/problematic requests?
- Was the issue explained clearly?
- Were helpful alternatives provided?
- Was the response professional?

**Score:** 0-1, where 1 = perfect edge case handling

**Use case:** Test cases with `should_flag_impossible` or `should_flag_conflict`

**Example output:**
```json
{
  "score": 0.95,
  "reasoning": "Agent correctly identified $20 budget as insufficient and offered helpful alternatives.",
  "correctly_identified_issues": ["Budget too low for travel"],
  "missed_issues": [],
  "appropriate_response": true,
  "suggestions_provided": true
}
```

### 5. Plan Type Compliance Evaluation

**What it checks:**
- If attractions-only requested, are flights/hotels excluded?
- If full plan requested, are all components present?
- Are there unnecessary components?

**Score:** 0-1, where 1 = perfect compliance with plan type

**Use case:** Test cases with `plan_type` field

**Example output:**
```json
{
  "score": 1.0,
  "reasoning": "User requested attractions only, and plan correctly excludes flights and accommodation while providing detailed itinerary.",
  "correctly_included": ["attractions", "restaurants", "reservations"],
  "correctly_excluded": ["flights", "hotels"],
  "incorrectly_included": [],
  "incorrectly_excluded": []
}
```

### 6. Special Requirements Evaluation

**What it checks:**
- Are accommodations appropriate for special needs (wheelchair, dietary, pets)?
- Are activities suitable?
- Were accommodations verified?
- Are emergency considerations addressed?

**Score:** 0-1, where 1 = perfect special requirements handling

**Use case:** Test cases with `special_requirements` field

**Example output:**
```json
{
  "score": 0.8,
  "reasoning": "All hotels verified for wheelchair access. Activities are accessible. Missing emergency medical facility info.",
  "requirements_met": ["wheelchair hotels", "accessible museums", "elevator access"],
  "requirements_missed": ["medical facilities nearby"],
  "verifications_included": true
}
```

### 7. Vague Request Handling Evaluation

**What it checks:**
- Did agent ask clarifying questions?
- Were destination suggestions appropriate?
- Were multiple options provided?
- Did agent explain why destinations fit criteria?

**Score:** 0-1, where 1 = perfect vague request handling

**Use case:** Test cases with `vague_` prefix in ID

**Example output:**
```json
{
  "score": 0.85,
  "reasoning": "Good destination suggestions for 'romantic beaches'. Asked about budget and region. Could ask about specific activities.",
  "clarifying_questions_asked": ["What's your budget?", "Which region do you prefer?"],
  "destination_suggestions": ["Maldives", "Bali", "Santorini"],
  "suggestions_appropriateness": "Very appropriate - all romantic beach destinations"
}
```

## Golden Dataset Structure

Test cases in `golden_dataset.json` follow this structure:

### Simple Test Case (Single Turn)
```json
{
  "id": "honeymoon_bali_14d",
  "description": "14-day luxury honeymoon to Bali",
  "user_query": "Plan a 14 day honeymoon trip to Bali for 2 people with a budget of $5000",
  "expected": {
    "destination": "Bali",
    "duration_days": 14,
    "budget_limit": 5000,
    "budget_currency": "USD",
    "trip_type": "honeymoon",
    "validation_rules": {
      "activities_should_include": ["romantic", "spa", "private", "dinner"],
      "accommodation_tier": "luxury"
    }
  }
}
```

### Multi-Turn Test Case
```json
{
  "id": "vague_romantic_honeymoon_beaches",
  "description": "Vague request - romantic honeymoon with beaches, no location specified",
  "conversation_turns": [
    {
      "turn": 1,
      "user_query": "I want to go to a romantic honeymoon with beaches",
      "agent_should_ask": ["destination_preferences", "budget", "duration"],
      "expected_behavior": "Agent should ask clarifying questions"
    },
    {
      "turn": 2,
      "user_response": "We have $4000 and can go for 10 days",
      "agent_should_ask": ["destination_region"],
      "expected_behavior": "Agent should suggest beach destinations"

  ],
  "final_expected": {
    "destination": ["Maldives", "Bali", "Phuket"],
    "duration_days": 10,
    "budget_limit": 4000
  }
}
```

### Edge Case Test Case
```json
{
  "id": "extreme_budget_impossible",
  "description": "Extreme budget - $20 for a week trip",
  "user_query": "I want to travel for a week with only $20",
  "expected": {
    "should_inform_impossible": true,

    "agent_should_suggest": ["increase_budget", "local_staycation"],
    "validation_rules": {
      "should_not_create_plan": true
    }
  }

```

## How to Use the Evaluation Framework

### Running Evaluations

```python
from tests.evaluations.eval_framework import evaluate_test_case
from app.graph import graph
import json

# Load test case
with open('tests/evaluations/golden_dataset.json') as f:
    test_cases = json.load(f)

# Run agent on test case
test_case = test_cases[0]
agent_output = graph.invoke({
    "messages": [test_case["user_query"]]
})

# Evaluate output
results = evaluate_test_case(agent_output, test_case)

print(f"Overall Score: {results['overall_score']}")
print(f"Passed: {results['passed']}")
print(json.dumps(results['evaluations'], indent=2))
```

### Interpreting Results

**Overall Score**: Average of all applicable evaluation scores (0-1 scale)
- **0.9-1.0**: Excellent - agent performed exceptionally
- **0.7-0.89**: Good - agent passed with minor issues
- **0.5-0.69**: Fair - significant improvements needed
- **< 0.5**: Poor - critical failures

**Passed**: Boolean (True if overall_score >= 0.7)

**Individual Evaluations**: Detailed scores and reasoning for each dimension

### Example Output
```json
{
  "test_id": "honeymoon_bali_14d",
  "test_description": "14-day luxury honeymoon to Bali",
  "overall_score": 0.875,
  "passed": true,

    "cost_realism": {
      "score": 0.9,
```

## Adding New Test Cases

### Step 1: Identify Test Scenario

Determine what you want to test:
- New trip type?
- New edge case?
- New special requirement?
- New conversation pattern?

### Step 2: Create Test Case

Add to `golden_dataset.json`:

```json
{
  "id": "your_test_id",
  "description": "Brief description",
  "user_query": "User's request" // OR conversation_turns for multi-turn
  "expected": {
    // What the agent should produce/do
  }
}
```

### Step 3: Define Expected Behavior

**For single-turn complete plans:**
```json
"expected": {
  "destination": "Paris",
  "duration_days": 7,
  "budget_limit": 3000,
  "trip_type": "family",
  "validation_rules": {
    "activities_should_include": ["museum", "park"],
    "accommodation_tier": "mid-range"
  }
}
```

**For multi-turn conversations:**
```json
"conversation_turns": [
  {
    "turn": 1,
    "user_query": "...",
    "agent_should_ask": ["budget", "duration"],
    "expected_behavior": "Description of what agent should do"
  }
]
```

**For edge cases:**
```json
"expected": {
  "should_flag_impossible": true,
  "agent_should_explain": "Why it's impossible",
  "agent_should_suggest": ["alternative1", "alternative2"]
}
```

### Step 4: Test It

Run the evaluation on your new test case and verify it catches the intended behavior.

## Best Practices

### For Developers

1. **Run evaluations before committing**: Ensure changes don't break existing functionality
2. **Add tests for new features**: When adding a feature, add corresponding test cases
3. **Check evaluation reasoning**: Don't just look at scores, read the reasoning to understand issues
4. **Iterate based on feedback**: Use evaluation insights to improve prompts and logic

### For Test Case Authors

1. **Be specific**: Define clear expected behaviors
2. **Cover edge cases**: Include boundary conditions and unusual requests
3. **Include variety**: Mix single-turn, multi-turn, and edge case tests
4. **Document intent**: Write clear descriptions explaining what you're testing

### For Prompt Engineers

1. **Test after prompt changes**: Always run evaluations after modifying prompts
2. **Use evaluation reasoning**: LLM explanations help identify prompt weaknesses
3. **Balance trade-offs**: Improving one dimension might affect another
4. **Validate improvements**: Ensure changes improve overall scores

## Troubleshooting

### Low Cost Realism Scores
- Check if destination cost guidelines need updating
- Verify accommodation tier mapping
- Review activity pricing databases

### Low Preference Alignment Scores
- Check if trip type definitions are clear
- Verify expected activities match trip type
- Review activity recommendation logic

### Low Multi-Turn Scores
- Ensure conversation context is preserved
- Check if supervisor asks appropriate questions
- Verify information extraction from user responses

### Low Edge Case Scores
- Implement detection logic for common edge cases
- Add validation before planning
- Improve error messages and alternative suggestions

### Batch Evaluation

```python
results = []
for test_case in test_cases:
    agent_output = run_agent(test_case)
    result = evaluate_test_case(agent_output, test_case)
    results.append(result)

# Aggregate scores

pass_rate = sum(r['passed'] for r in results) / len(results)
```
