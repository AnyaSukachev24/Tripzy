# The Graph Architecture

We will use a Supervisor Architecture (also known as a Router). The "Supervisor" acts as the traffic controller, maintaining the context of which customer we are currently helping and routing commands to the right specialist.

## 1. The Nodes (Sub-Agents)

### Supervisor (The Router)
- **The Supervisor now has 3 Modes**:
  - **Router Mode**: "You asked for X, so I will send you to Agent Y."
  - **Question Mode**: "I need more info from you." (e.g., "Which Sarah? There are two.")
  - **Output:** A focused `supervisor_instruction` for the worker
  - **Safety Mode**: "I am about to do something big. Do you approve?"
- **Logic**:
  - If the user asks "Find a customer..." $\rightarrow$ Route to **CRM_Retriever**.
  - If the user asks "Where should they go?" or "Plan a trip..." $\rightarrow$ Route to **Trip_Planner**.
  - If the user asks "Book this" or "Send email" $\rightarrow$ Route to **Action_Executor**.
- **Key Context**: It checks if a `active_customer` is already selected in the State.

## The Sub-Agents 
- They just do work and return data to the Supervisor.
- The Supervisor packages that data into a nice chat message for the user.

### CRM_Retriever (The Hunter)
- **Role**: Finds customers in the Vector DB.
- **Capabilities**:
  - **Semantic Search**: "Find families wanting to go to Disney" (Vector match).
  - **Filter Search**: "Show me urgent requests" (Metadata filter).
  - **Lookup**: "Get customer ID 42".
- **Outcome**: It populates the `active_customer` field in the State.

### Trip_Planner (The Brain)
- **Role**: Matches the `active_customer`'s preferences to your Destination Database.
- **Capabilities**: Reads the customer's unstructured summary (e.g., "loves history and wine") and queries the Destination DB to find the best match (e.g., "Tuscany").
- **Outcome**: Generates a structured `trip_plan` in the State.

### Action_Executor (The Doer)
- **Role**: Interacts with the outside world (Mock Tools).
- **Capabilities**: Booking flights, sending emails, generating files.
  - IF `awaiting_approval` is True: Check `user_query` for "Yes".
  - IF `awaiting_approval` is False: Read `supervisor_instruction`, call LLM, return `pending_action`.
- **Outcome**: Updates the system status (e.g., changes customer status from "Urgent" to "Replied").

---

## 2. The State (Memory)

The State is the shared memory passed between these nodes. It must track who we are talking about and what we have planned so far.

### State Schema
- **user_query**: `String` (The raw input from the travel agent).
- **active_customer**: `JSON Object` (The full profile of the customer currently being helped. If Null, the assistant knows it needs to find someone first).
- **trip_plan**: `JSON Object` (The proposed destination, flight, and hotel details generated by the Planner).
- **steps**: `List` (REQUIRED: The audit log of every tool call for grading).
- **pending_action**: `JSON` (e.g., {"tool": "book_flight", "params": {...}}). This holds the action in "Limbo" until approved.
- **messages**: `List` (Chat history).
- **awaiting_approval**: `Boolean` (True/False).
- **next_node**: `String` (The next node to route to).

## 3. The Tools Breakdown

Here are the specific tools each agent needs access to.

### Assigned to CRM_Retriever
- **`search_customers_vector(query: str, k: int)`**
  - **Description**: Embeds the query and searches the Vector DB (Pinecone) for semantically similar customer summaries.
  - **Use Case**: "Find customers who want a beach vacation."
- **`filter_customers(status: str, urgency: bool)`**
  - **Description**: Standard SQL-style filter on the metadata.
  - **Use Case**: "Who is waiting for a reply?" or "Show me urgent cases."
- **`get_customer_by_id(id: str)`**
  - **Description**: Direct retrieval.

### Assigned to Trip_Planner
- **`search_destinations(tags: list, budget_tier: str)`**
  - **Description**: Queries your Destinations Database.
  - **Use Case**: Finding "Tuscany" based on "History + Wine" tags.
- **`check_availability(destination: str, dates: str)`**
  - **Description**: (Mock) Checks if flights/hotels exist for the requested dates.

### Assigned to Action_Executor
- **`book_service(service_type: str, details: dict)`**
  - **Description**: (Mock) "Books" the flight/hotel/restaurant. Returns a confirmation code.
- **`send_email(to: str, subject: str, body: str)`**
  - **Description**: (Mock) Simulates sending the proposal to the client.
- **`generate_itinerary_file(plan_data: dict)`**
  - **Description**: Creates the formal file (PDF/Doc) mentioned in your requirements.

## 4. Example Data Flow

**Scenario**: The human agent says: "Find Sarah Jenkins and book her a flight to her destination."

1. **Supervisor**: Sees "Find Sarah". Routes to $\rightarrow$ **CRM_Retriever**.
2. **CRM_Retriever**: Calls `search_customers_vector("Sarah Jenkins")`.
   - **Result**: Finds Sarah's JSON profile (which includes destination: "Orlando").
   - **State Update**: Sets `active_customer = {Sarah's Data}`.
3. **Supervisor**: Sees we have an active customer and the user asked to "book". Routes to $\rightarrow$ **Trip_Planner** (to verify details) or directly to **Action_Executor**.
4. **Action_Executor**: Calls `book_service("flight", destination="Orlando")`.
   - **Result**: "Booking Confirmed: FL-123".
   - **State Update**: Logs the step.
5. **End**: Returns confirmation to the user.
